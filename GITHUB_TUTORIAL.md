# GitHub 与 IDE 源代码管理新手教程

你好！这是一份专为你准备的 Git 和 GitHub 使用指南。作为初学者，你不需要死记硬背复杂的命令，只需要理解核心流程，并学会使用 IDE（代码编辑器）自带的可视化工具即可。

## 🌟 第一部分：核心概念（通俗版）

把 Git 想象成一个**时光机**和**云同步盘**的结合体。

1.  **Git (工具)**：安装在你电脑上的时光机软件，负责记录代码的每一次修改。
2.  **GitHub (网站)**：一个云端硬盘，用来存放你的代码备份，并展示给别人看。
3.  **仓库 (Repository)**：你的项目文件夹。
4.  **提交 (Commit)**：**关键概念！** 相当于游戏里的“手动存档”。你写完一部分代码，觉得没问题了，就 Commit 一次。
5.  **推送 (Push)**：把本地的“存档”上传到 GitHub 服务器。

## 🚀 第二部分：IDE (Trae/VS Code) 操作指南

你不需要敲黑黑的命令行窗口，IDE 已经把功能集成得很好了。请看左侧侧边栏，找到那个**“源代码管理”图标**（通常是一个分叉的连接线图标）。

### 日常开发标准流程（每天都要做！）

#### 1. 写代码
当你修改并保存 (`Ctrl+S`) 任何文件后，源代码管理图标上会出现一个数字气泡，提示你有几个文件发生了变化。

#### 2. 暂存 (Stage) —— "挑选要存档的文件"
*   点击源代码管理图标。
*   在 **"更改 (Changes)"** 列表中，找到你修改的文件。
*   点击文件旁边的 **`+` (加号)**。
*   **效果**：文件会从 "更改" 移动到 **"暂存的更改 (Staged Changes)"**。这意味着你告诉 Git：“这几个文件的修改，我要打包存个档”。

#### 3. 提交 (Commit) —— "确认存档"
*   在输入框中填写 **提交信息 (Message)**。
    *   ❌ 错误示范：`update`, `111`, `fix`
    *   ✅ 正确示范：`修复了图片识别的bug`，`新增了GitHub教程文档`，`更新了README说明`
*   点击 **"提交 (Commit)"** 按钮（通常是蓝色按钮或 `√` 图标）。
*   **效果**：代码已保存到本地历史记录，现在的状态是安全的。

#### 4. 同步 (Sync/Push) —— "上传云端"
*   提交完成后，按钮通常会变成 **"同步更改 (Sync Changes)"** 或显示一个向上的箭头。
*   点击它。
*   **效果**：你的本地存档会被上传到 GitHub 网站。此时你可以去 GitHub 网页上刷新，就能看到最新的代码了。

## 🔧 第三部分：常用场景与对策

### Q1: 我想看我以前写过什么代码？
*   **在 IDE 中**：通常安装 "Git Graph" 或 "Git History" 插件，可以看到像地铁线路图一样的历史记录。
*   **在 GitHub 网页**：点击项目主页的 **"Commits"**（通常是一个时钟图标），你可以看到每一次提交的记录，点击进去能看到具体改了哪一行代码（红色是删除，绿色是新增）。

### Q2: 我把代码改坏了，想回退怎么办？
*   右键点击该文件，选择 **"Discard Changes" (放弃更改)** 或 **"撤销更改"**（通常是一个回转箭头图标）。
*   **警告**：这会把你最后一次保存后的修改全部清空，恢复到上一次 Commit 的状态。

### Q3: 为什么报错说 "Conflict" (冲突)？
*   **原因**：通常是因为你和别人修改了同一个文件的同一行代码，Git 不知道该听谁的。
*   **解决**：IDE 会在代码里用颜色标出冲突的地方。你需要手动选择保留哪一段代码，或者把两段都留着，改好后再次提交。

## 📝 常用 Git 命令速查（备用）

虽然推荐用 IDE 按钮，但了解这些命令能让你更专业：

```bash
# 1. 初始化仓库（新项目第一次用）
git init

# 2. 查看状态（看哪些文件改了）
git status

# 3. 添加文件到暂存区（对应点 + 号）
git add . 

# 4. 提交存档（对应点提交按钮）
git commit -m "这里写你的提交备注"

# 5. 推送云端（对应点同步按钮）
git push origin main

# 6. 拉取更新（把云端的最新代码下载下来）
git pull origin main
```

## 💡 给初学者的建议

1.  **多提交 (Commit often)**：不要写了一整天代码才提交一次。每完成一个小功能（比如写好了一个函数，或者改好了一个配置），就提交一次。这样如果出错了，你回退的损失很小。
2.  **写好备注**：提交信息是写给未来的自己看的。
3.  **每天下班/放学前 Push**：确保代码在云端有备份。
